# Bundle Availability and Asymmetric DLQ Consumption

Status: Accepted. Renumbered from duplicate ADR 0034.

`revisions.bundle_status` is a four-value enum (`pending`, `ready`, `failed`, `disabled`) and the `bundle-generate` DLQ is consumed by `jobs` to flip `failed` when a message dead-letters. The `safety-scan` and `byte-purge` DLQs are intentionally not consumed. The governing principle: a DLQ gets a consumer only when terminal failure must update Postgres for product correctness; otherwise the DLQ is a holding area for operator triage.

## Consequences

- **`revisions.bundle_status` enum:** `disabled` (set by `api` at **Publish** time when **Usage Policy** prohibits a **Bundle**), `pending` (enqueued; covers queued and actively generating), `ready` (bytes at the deterministic R2 key from ADR 0021, with `bundle_size_bytes` populated), `failed` (DLQ consumer recorded a terminal failure). Plus `bundle_status_updated_at TIMESTAMPTZ`.
- **Agent View shape for `bundle`:** `status` is always present; `url` and `size_bytes` only on `ready`; `retry_after_seconds` only on `pending` as a polling hint. No `error` field on `failed` — internal detail does not leak through the Agent View; operators consult op logs.
- **`bundle-generate-dlq` consumer.** Lives in `jobs`, batch size 10. Decodes the original message, extracts `revision_id`, and runs `runCommand` with `actor_id='bundle_generate'`, `operation='bundle.mark_failed'`, `key=revision_id` to UPDATE `bundle_status='failed'` with an empty audit list (per ADR 0019, bundle status changes do not emit audit events). Emits a structured op log at error level with `final_failure=true`.
- **`safety-scan-dlq` and `byte-purge-dlq` have no consumers.** A scan failure leaves existing **Safety Warnings** untouched and writes no replacement, so no Postgres state needs flipping. A byte-purge failure leaves R2 bytes in place; the `bytes_purge_enqueued_at` column already indicates "attempted." Both DLQs alert on `depth > 0` for 5 minutes; operator triage drives recovery.
- **Retry policy** is asymmetric: `bundle-generate` allows 5 attempts because the work is IO-heavy and benefits from transient-failure backoff; `safety-scan` and `byte-purge` stay at the Cloudflare default of 3 because scan failures are typically deterministic and purge work is idempotent enough that further retries waste compute.
- **`failed` is terminal for the MVP.** Automatic re-enqueue from a cron sweep is not provided. A future workspace-member-initiated regenerate surface (via `api`) will need either a distinct `operation` key (`bundle.regenerate:{revision_id}:{nonce}`) or explicit invalidation of the prior `idempotency_records` row before re-enqueue. The choice can be made when that surface lands.
- **Race short-circuit.** The `bundle-generate` handler reads `bundle_status` and parent `artifacts.status` before generating. If status is already `ready` or `disabled`, or if the revision is `retained` or the artifact `deleted`, the handler returns idempotently without work. This handles the publish-then-immediately-retention race.
- **`disabled` is set by `api`, never by `jobs`.** It is written inside the **Publish** `runCommand` transaction when **Usage Policy** rejects bundle generation for the **Revision**. `jobs` never transitions a **Revision** to `disabled`.
