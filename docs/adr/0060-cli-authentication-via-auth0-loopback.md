# CLI Authentication via WorkOS Loopback

Status: Accepted. Provider decided as **WorkOS** per [ADR 0068](./0068-workos-authkit-for-web-app-auth.md); this replaces the Auth0 framing this ADR originally carried (the filename is kept by repo convention). Recorded in the Phase 2/3 reconciliation (decision D2). The CLI-vs-dashboard scope boundary, previously deferred to implementation, is now resolved. Implemented as Option B (mint a scoped API key, discard the WorkOS token); see **Implementation (as built)**, which supersedes the token-as-credential and per-client scope-ceiling mechanisms described later in this ADR. Verified end-to-end against preview 2026-05-24.

The CLI gains an `agent-paste login` subcommand that runs a WorkOS authorization-code-with-PKCE flow via a localhost loopback redirect, against the same WorkOS project the dashboard uses (ADR 0068). The resulting access and refresh tokens replace `AGENT_PASTE_API_KEY` as the default CLI credential; the env-var **API Key** path stays for headless and CI use. `apps/api` verifies CLI access tokens with the same JWKS + `client_id` path it already uses for dashboard tokens (ADR 0068). A Workspace Member acting through the CLI is strictly less powerful than the same Workspace Member acting through the dashboard; that boundary is expressed by registering the CLI as a **dedicated WorkOS OAuth client** and having `api` apply a **per-client scope ceiling** keyed on the token's originating client (resolved below).

## Implementation (as built)

This section supersedes the token-as-credential mechanism and the per-client scope-ceiling code described in the Considered Options and Consequences below. The flow that shipped is **Option B: mint a scoped API key**.

- **Mint, do not store the WorkOS token.** `agent-paste login` runs loopback authorization-code+PKCE against a dedicated WorkOS **Public OAuth (Connect)** application, exchanges the code for a short-lived access token, and uses that token for exactly one call: `POST /v1/web/keys` to mint an API key. The minted key is the durable CLI credential; the WorkOS access token is discarded immediately after minting. There is no refresh-token storage and no refresh machinery (the "Refresh" consequence below does not apply).
- **Scope ceiling realized by the key model, not by per-client code.** Minted API keys are hardcoded to `["publish", "read"]` (`packages/db/src/repository/shared.ts`) and never `admin`. The CLI is therefore structurally less powerful than the dashboard with no per-client scope-intersection code in `api`. The "Scope boundary (resolved)" consequence below is realized this way; no Member-Only-Scope stripping logic exists.
- **Public OAuth (Connect) app + wildcard loopback.** The CLI uses its own WorkOS Public OAuth (Connect) `client_id` (public identifier, distinct from the dashboard AuthKit `client_id`) as the authorize/token-exchange client; it lives only in the CLI config (`apps/cli/src/config.ts`), not in `api`. The registered redirect is the wildcard `http://127.0.0.1:*/callback`, so the CLI binds an ephemeral loopback port per login. Connect access tokens are verified at the app's AuthKit-domain JWKS (`https://<subdomain>.authkit.app/oauth2/jwks`) with issuer `https://<subdomain>.authkit.app`, not the dashboard's `/sso/jwks/{client_id}` at `api.workos.com`.
- **Token pinned by environment audience, not the OAuth app id.** A WorkOS Connect access token's `aud` is the **environment OIDC client**, and it carries no `client_id`/`azp` claim — so the token cannot be pinned to the CLI OAuth app's `client_id` via any JWT claim. The verifiable pin is issuer + JWKS signature + the environment audience, set in `api` as `WORKOS_CLI_AUDIENCE` (`apps/api/wrangler.jsonc`, preview/production). This corrects the original design's claim (in the Consequences below) that `aud = cli_client_id` discriminates a CLI token.
- **CLI token confined to the mint path.** `api` accepts a CLI WorkOS token on `web.apiKeys.create` only. Every other `workos_access_token` route stays dashboard-only. The audience pin (`WORKOS_CLI_AUDIENCE`) is added only for the key-mint route; when unset, behavior is exactly dashboard-only.
- **JIT provisioning = CLI-first signup.** The key-mint route is the CLI's only entry point, so on first contact it provisions a Personal Workspace, Workspace Member, and default key (the same `resolveWebMember` path as the dashboard callback, ADR 0055) for a brand-new WorkOS user, then mints the CLI key. Every other `workos_access_token` route still requires an already-provisioned member.
- **Storage shape.** The stored credential is `{ api_key, public_id, workspace_id, member_email }` (the minted key, not WorkOS tokens). macOS uses the login Keychain via `security`; other platforms use `~/.config/agent-paste/credentials.json` at mode `0600` under a `0700` parent.
- **Precedence unchanged.** `AGENT_PASTE_API_KEY` wins over the stored key; a one-line stderr note names the winner when both are present.

## Considered Options

- **Status quo (API Key only).** Forces every CLI user to mint and rotate a long-lived secret manually, then keep it in shell history, dotfiles, or env-var managers. The credential never expires unless someone remembers to set an **Expiration**. Rejected because the long-lived secret on a developer machine is exactly the artifact this ADR exists to eliminate.
- **Device code flow.** Works in SSH sessions and remote dev containers; adds a paste-the-code step and a second flow to implement. Loopback covers the dominant case (interactive laptop); device code is deferred until a real headless-interactive use case appears.
- **A second IdP for the CLI (e.g. keep Auth0 for CLI, WorkOS for web).** Two providers to operate, two sets of secrets, two failure modes. Rejected: one IdP for all human auth is the point of consolidating on WorkOS in ADR 0068.
- **WorkOS loopback authorization-code with PKCE (chosen).** Standard pattern for CLI auth across `gh`, `vercel`, `supabase`. Browser-mediated, short-lived access tokens, refresh tokens for sustained sessions, no manual code entry. Reuses the dashboard's WorkOS project and the existing `api` JWKS verifier. ADR 0068 already notes WorkOS supports loopback PKCE.

## Consequences

- **Command.** `agent-paste login` opens the default browser to the WorkOS AuthKit authorization endpoint with `response_type=code`, a PKCE `code_challenge`, `redirect_uri=http://127.0.0.1:{ephemeral_port}/callback`, and `state`. The CLI runs a one-shot loopback HTTP listener bound to `127.0.0.1` on an ephemeral port it picks at start so concurrent logins do not collide. The listener returns a static success page and shuts down on first callback or `state` mismatch. The CLI exchanges the code for tokens at the WorkOS token endpoint with the PKCE `code_verifier`; no client secret is held on the machine.
- **Same WorkOS project, dedicated OAuth client.** The CLI authenticates against the same WorkOS project as the dashboard so identities and Workspace Member provisioning (ADR 0055) are shared. The CLI is registered as its own **WorkOS OAuth (Connect) application** — distinct `client_id` from the dashboard's AuthKit client — with `http://127.0.0.1` loopback redirects allowed (PKCE-only, public client, no secret on the machine). A loopback authorization-code+PKCE exchange against this client yields an access token carrying `aud = <cli_client_id>` in the body, which is the in-token discriminator `api` reads to recognize a CLI token. The dashboard's AuthKit session token has no `aud` and is recognized by its session shape; the two clients also have separate JWKS, so a CLI token can never validate as a dashboard token even setting `aud` aside.
- **Token verification.** `apps/api` verifies CLI access tokens through the JWKS + `client_id` check established in ADR 0068, selecting the CLI client's JWKS. No Auth0-style custom-audience verifier is introduced; the `audience=https://api.agent-paste.sh/v1/cli` string from the original ADR is dropped. The `aud` claim here is WorkOS's native OAuth-client audience (the `client_id`), not an app-defined API audience.
- **Scope boundary (resolved).** The CLI is strictly less powerful than the dashboard: it gets `write read share` and never the **Member-Only Scopes** (`manage_keys`, `manage_workspace`, `read_audit`). WorkOS RBAC cannot express this — `role`/`permissions` claims are scoped to the user's **organization membership**, so the same user gets the identical claims whether they log in via CLI or dashboard, and the JWT template is per-environment with no handle for the requesting client. The boundary is therefore enforced **server-side as a per-client scope ceiling**: `api` resolves the originating client (CLI via `aud = cli_client_id`, dashboard via the AuthKit session shape), loads the member's base **Scopes** from the `workspace_members.scopes` column as today, and **intersects** them with that client's ceiling — the CLI ceiling strips the three Member-Only Scopes, the dashboard ceiling allows all. The scope _source_ stays the database row, not the token; WorkOS RBAC remains an optional future refinement and is not required to enforce this boundary. This re-qualifies ADR 0034's "Workspace Member is implicitly granted every Scope" rule: the implicit grant is the dashboard ceiling, and the CLI ceiling is the carve-out — the same invariant the Auth0 audience exception protected, now keyed on the originating WorkOS client instead of an Auth0 audience.
- **Storage.**
  - macOS: `security add-generic-password -s "agent-paste" -a "{workos_user_id}" -w "{json_blob}" -U` writes to the user's login Keychain. Read with `security find-generic-password -s "agent-paste" -a "{workos_user_id}" -w`. Multiple identities on the same Mac coexist via distinct `account` values.
  - Linux/Windows: `~/.config/agent-paste/credentials.json` created with mode `0600`, parent directory mode `0700`. JSON shape `{ access_token, refresh_token, expires_at, sub }`, the same shape `web` seals into `__agp_session` per [ADR 0059](./0059-web-app-session-and-auth-forwarding-to-api.md), minus the seal because the file is on a trusted machine.
  - No keyring integration on Linux in v1. libsecret/Secret Service support is a follow-up; the file path covers the typical Linux developer setup without adding a system dependency to the CLI.
- **Refresh.** Before every `api` call, the CLI checks `expires_at`. When within 60 seconds of expiry, the CLI performs a WorkOS refresh-token grant, rewrites storage, and proceeds. Concurrent CLI invocations may race on the rewrite; accepted, the loser refreshes again on its next call and one refresh-token chain wins. No file locking, no daemon.
- **Logout.** `agent-paste logout` deletes the Keychain entry or the credentials file. No remote revocation hop in v1; the access token expires within minutes and the refresh token is no longer reachable from this machine. A future revoke endpoint can be added without changing the storage shape.
- **Precedence.** When `AGENT_PASTE_API_KEY` is set, the CLI uses it directly and skips Keychain/file lookup, so CI agents always behave predictably. The CLI emits a one-line note to stderr when both are present, naming which credential it picked.
- **`whoami` output** is identical across credential classes: Workspace name, Workspace Member identity (when session-authenticated) or **API Key** name and **Workspace ID** (when key-authenticated), and the resolved **Scopes**. Tooling can ignore the credential class and read scopes uniformly.
- **CLI README update.** The "Authenticate" section gains `agent-paste login` as the primary path with `AGENT_PASTE_API_KEY` documented as the CI/headless alternative.
- **CONTEXT.md.** The **CLI** glossary entry already names "an OIDC loopback flow for humans (provider chosen when CLI human auth lands)"; that provider is now WorkOS. The **Scope** implicit-grant note should record that the implicit full grant is the dashboard ceiling and the CLI surface gets a reduced ceiling (no Member-Only Scopes), keyed on the originating WorkOS client. No new glossary noun.
