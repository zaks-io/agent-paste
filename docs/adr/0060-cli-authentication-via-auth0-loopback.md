# CLI Authentication via WorkOS Loopback

Status: Accepted. Provider decided as **WorkOS** per [ADR 0068](./0068-workos-authkit-for-web-app-auth.md); this replaces the Auth0 framing this ADR originally carried (the filename is kept by repo convention). Recorded in the Phase 2/3 reconciliation (decision D2). Not yet implemented; today the CLI authenticates with `AGENT_PASTE_API_KEY` only.

The CLI gains an `agent-paste login` subcommand that runs a WorkOS AuthKit authorization-code-with-PKCE flow via a localhost loopback redirect, against the same WorkOS project the dashboard uses (ADR 0068). The resulting access and refresh tokens replace `AGENT_PASTE_API_KEY` as the default CLI credential; the env-var **API Key** path stays for headless and CI use. `apps/api` verifies CLI access tokens with the same JWKS + `client_id` path it already uses for dashboard tokens (ADR 0068). A Workspace Member acting through the CLI is intended to be strictly less powerful than the same Workspace Member acting through the dashboard; how that boundary is expressed in WorkOS terms is the one open implementation detail (see below).

## Considered Options

- **Status quo (API Key only).** Forces every CLI user to mint and rotate a long-lived secret manually, then keep it in shell history, dotfiles, or env-var managers. The credential never expires unless someone remembers to set an **Expiration**. Rejected because the long-lived secret on a developer machine is exactly the artifact this ADR exists to eliminate.
- **Device code flow.** Works in SSH sessions and remote dev containers; adds a paste-the-code step and a second flow to implement. Loopback covers the dominant case (interactive laptop); device code is deferred until a real headless-interactive use case appears.
- **A second IdP for the CLI (e.g. keep Auth0 for CLI, WorkOS for web).** Two providers to operate, two sets of secrets, two failure modes. Rejected: one IdP for all human auth is the point of consolidating on WorkOS in ADR 0068.
- **WorkOS loopback authorization-code with PKCE (chosen).** Standard pattern for CLI auth across `gh`, `vercel`, `supabase`. Browser-mediated, short-lived access tokens, refresh tokens for sustained sessions, no manual code entry. Reuses the dashboard's WorkOS project and the existing `api` JWKS verifier. ADR 0068 already notes WorkOS supports loopback PKCE.

## Consequences

- **Command.** `agent-paste login` opens the default browser to the WorkOS AuthKit authorization endpoint with `response_type=code`, a PKCE `code_challenge`, `redirect_uri=http://127.0.0.1:{ephemeral_port}/callback`, and `state`. The CLI runs a one-shot loopback HTTP listener bound to `127.0.0.1` on an ephemeral port it picks at start so concurrent logins do not collide. The listener returns a static success page and shuts down on first callback or `state` mismatch. The CLI exchanges the code for tokens at the WorkOS token endpoint with the PKCE `code_verifier`; no client secret is held on the machine.
- **Same WorkOS project, distinct client.** The CLI authenticates against the same WorkOS project as the dashboard so identities and Workspace Member provisioning (ADR 0055) are shared. A dedicated WorkOS client/application is registered for the CLI with `http://127.0.0.1` loopback redirects allowed (PKCE-only, public client). Whether CLI and dashboard tokens are told apart by separate `client_id`s or by another claim is settled at build time (see open detail).
- **Token verification.** `apps/api` verifies CLI access tokens through the JWKS + `client_id` check established in ADR 0068. No new audience-based verifier is introduced; the Auth0-specific `audience=https://api.agent-paste.sh/v1/cli` string from the original ADR is dropped.
- **Scope boundary (open implementation detail).** The CLI must remain strictly less powerful than the dashboard: it gets `write read share` and never the **Member-Only Scopes** (`manage_keys`, `manage_workspace`, `read_audit`). Under Auth0 this was an audience carve-out amending ADR 0034. WorkOS access tokens do not carry a custom audience the same way, so the mechanism must be re-expressed when CLI auth is built: candidates are a dedicated CLI WorkOS client whose tokens `api` maps to the reduced scope set, or an explicit scope claim. ADR 0034's "Workspace Member is implicitly granted every Scope" rule must be re-qualified against whatever signal distinguishes a CLI token from a dashboard cookie session. This is the one decision deferred to implementation.
- **Storage.**
  - macOS: `security add-generic-password -s "agent-paste" -a "{workos_user_id}" -w "{json_blob}" -U` writes to the user's login Keychain. Read with `security find-generic-password -s "agent-paste" -a "{workos_user_id}" -w`. Multiple identities on the same Mac coexist via distinct `account` values.
  - Linux/Windows: `~/.config/agent-paste/credentials.json` created with mode `0600`, parent directory mode `0700`. JSON shape `{ access_token, refresh_token, expires_at, sub }`, the same shape `web` seals into `__agp_session` per [ADR 0059](./0059-web-app-session-and-auth-forwarding-to-api.md), minus the seal because the file is on a trusted machine.
  - No keyring integration on Linux in v1. libsecret/Secret Service support is a follow-up; the file path covers the typical Linux developer setup without adding a system dependency to the CLI.
- **Refresh.** Before every `api` call, the CLI checks `expires_at`. When within 60 seconds of expiry, the CLI performs a WorkOS refresh-token grant, rewrites storage, and proceeds. Concurrent CLI invocations may race on the rewrite; accepted, the loser refreshes again on its next call and one refresh-token chain wins. No file locking, no daemon.
- **Logout.** `agent-paste logout` deletes the Keychain entry or the credentials file. No remote revocation hop in v1; the access token expires within minutes and the refresh token is no longer reachable from this machine. A future revoke endpoint can be added without changing the storage shape.
- **Precedence.** When `AGENT_PASTE_API_KEY` is set, the CLI uses it directly and skips Keychain/file lookup, so CI agents always behave predictably. The CLI emits a one-line note to stderr when both are present, naming which credential it picked.
- **`whoami` output** is identical across credential classes: Workspace name, Workspace Member identity (when session-authenticated) or **API Key** name and **Workspace ID** (when key-authenticated), and the resolved **Scopes**. Tooling can ignore the credential class and read scopes uniformly.
- **CLI README update.** The "Authenticate" section gains `agent-paste login` as the primary path with `AGENT_PASTE_API_KEY` documented as the CI/headless alternative.
- **CONTEXT.md.** The **CLI** glossary entry already names "an OIDC loopback flow for humans (provider chosen when CLI human auth lands)"; that provider is now WorkOS. The **Scope** implicit-grant qualification follows the open implementation detail above. No new glossary noun.
