# CLI Authentication via Auth0 Loopback

The CLI gains an `agent-paste login` subcommand that runs an Auth0 authorization-code-with-PKCE flow against the existing Auth0 tenant via a localhost loopback redirect. The resulting access and refresh tokens replace `AGENT_PASTE_API_KEY` as the default CLI credential; the env-var **API Key** path stays for headless and CI use. CLI access tokens are minted under a new audience `https://api.agent-paste.sh/v1/cli` and carry explicit **Scopes** (`write read share`), never **Member-Only Scopes**. This means a Workspace Member acting through the CLI is strictly less powerful than the same Workspace Member acting through the dashboard, which is the intended security boundary.

## Considered Options

- **Status quo (API Key only).** Forces every CLI user to mint and rotate a long-lived secret manually, then keep it in shell history, dotfiles, or env-var managers. The credential never expires unless someone remembers to set an **Expiration**. Rejected because the long-lived secret on a developer machine is exactly the artifact this ADR exists to eliminate.
- **Device code flow.** Works in SSH sessions and remote dev containers; adds a paste-the-code step and a second flow to implement. Loopback covers the dominant case (interactive laptop); device code is deferred until a real headless-interactive use case appears.
- **Loopback authorization-code with PKCE (chosen).** Standard pattern for CLI auth across `gh`, `vercel`, `supabase`. Browser-mediated, short-lived access tokens, refresh tokens for sustained sessions, no manual code entry.
- **Mint a fresh API Key per login as the "session" credential.** Considered because it reuses the existing `ap_pk_...` machinery end-to-end. Rejected because it turns the OAuth flow into a thin secret-minting layer — refresh becomes API-Key-rotation, and the dashboard's API Key list fills up with `cli:host-...` entries. The Auth0 JWT path established by ADR 0059 is the right shape; CLI just adopts it locally.

## Consequences

- **Command.** `agent-paste login` opens the default browser to `https://{auth0-tenant}/authorize?response_type=code&code_challenge=...&redirect_uri=http://127.0.0.1:{ephemeral_port}/callback&audience=https://api.agent-paste.sh/v1/cli&scope=write+read+share+offline_access&state=...`. The CLI runs a one-shot loopback HTTP listener bound to `127.0.0.1` on an ephemeral port that the listener picks at start so multiple concurrent logins do not collide. The listener returns a static success page and shuts down on first callback or `state` mismatch.
- **New audience.** `https://api.agent-paste.sh/v1/cli` distinguishes CLI tokens from dashboard tokens (`https://api.agent-paste.sh/v1`) at `api`'s auth middleware. The audience drives the implicit-grant carve-out documented in [ADR 0034](./0034-unified-scope-model-across-actors.md).
- **Scope set.** `write read share offline_access`. The CLI cannot request **Member-Only Scopes** (`manage_keys`, `manage_workspace`, `read_audit`) — those remain dashboard-only. `offline_access` enables refresh-token issuance.
- **Auth0 application.** A new Auth0 "Native" application named `agent-paste CLI`. Allowed callback URLs: `http://127.0.0.1` (Auth0 accepts any port when the host is `127.0.0.1` for native applications). PKCE required; no client secret.
- **Storage.**
  - macOS: `security add-generic-password -s "agent-paste" -a "{auth0_sub}" -w "{json_blob}" -U` writes to the user's login Keychain. Read with `security find-generic-password -s "agent-paste" -a "{auth0_sub}" -w`. Multiple identities on the same Mac coexist via distinct `account` values.
  - Linux/Windows: `~/.config/agent-paste/credentials.json` created with mode `0600`, parent directory mode `0700`. JSON shape `{ access_token, refresh_token, expires_at, sub }` — the same shape `web` seals into `__agp_session` per [ADR 0059](./0059-web-app-session-and-auth-forwarding-to-api.md), minus the seal because the file is on a trusted machine.
  - No keyring integration on Linux in v1. libsecret/Secret Service support is a follow-up; the file path covers the typical Linux developer setup without adding a system dependency to the CLI.
- **Refresh.** Before every `api` call, the CLI checks `expires_at`. When within 60 seconds of expiry, the CLI performs an Auth0 refresh-token grant, rewrites storage, and proceeds. Concurrent CLI invocations may race on the rewrite — accepted; the loser sees its in-memory token, refreshes again on its next call, and one of the two refresh-token-replacement chains wins. No file locking, no daemon.
- **Logout.** `agent-paste logout` deletes the Keychain entry or the credentials file. No remote revocation hop in v1; the access token expires within minutes and the refresh token is no longer reachable from this machine. A future revoke endpoint can be added without changing the storage shape.
- **Precedence.** When `AGENT_PASTE_API_KEY` is set, the CLI uses it directly and skips Keychain/file lookup. The CLI prefers the explicit env-var even if a stored session exists, so CI agents always behave predictably. The CLI emits a one-line note to stderr when both are present, naming which credential it picked.
- **`whoami` output** is identical across credential classes: Workspace name, Workspace Member identity (when session-authenticated) or **API Key** name and **Workspace ID** (when key-authenticated), and the resolved **Scopes**. Tooling can ignore the credential class and read scopes uniformly.
- **CLI README update.** The "Authenticate" section gains `agent-paste login` as the primary path with `AGENT_PASTE_API_KEY` documented as the CI/headless alternative. The "encodes its Workspace" note about API Keys remains accurate for that path; for sessions the Workspace is resolved from the JWT `sub`.
- **ADR 0034 amendment.** The "**Workspace Member** is implicitly granted every **Scope**" rule narrows: it applies only to tokens with `audience=https://api.agent-paste.sh/v1` (dashboard cookie sessions via [ADR 0059](./0059-web-app-session-and-auth-forwarding-to-api.md)). Tokens with `audience=https://api.agent-paste.sh/v1/cli` use their `scope` claim literally. See [ADR 0034](./0034-unified-scope-model-across-actors.md) for the amended consequence.
- **CONTEXT.md amendment.** The **Scope** glossary entry and the relationships block are refined so the implicit-grant rule is qualified by authentication audience rather than asserted universally. No new glossary noun.
- **What this ADR does not introduce.** No new domain concept for the local credential store. The artifact on disk is implementation detail of the CLI, not a glossary term. The user-facing identity remains **Workspace Member**.
