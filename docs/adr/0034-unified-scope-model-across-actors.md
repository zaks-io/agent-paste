# Unified Scope Model Across Actors

**Scopes** apply uniformly to every actor that can call `api`: a **Workspace Member** is implicitly granted every **Scope** _when authenticated for direct workspace control through the dashboard_, and an **API Key** holds a named subset. A **Member-Only Scope** subset (`manage_keys`, `manage_workspace`, `read_audit`) authorizes operations that only a **Workspace Member** can perform; member-only **Scopes** cannot be granted to an **API Key**. This collapses authorization to a single `requireScopes` middleware while preserving the asymmetry between human and agent authority that ADR 0015 establishes.

## Surface-scoped exception

> **Provider note (WorkOS, ADR 0068 and ADR 0061).** The Auth0 custom-`aud` framing below is superseded. A WorkOS AuthKit dashboard session is the direct workspace-control surface and receives the implicit full member grant. `agent-paste login` uses WorkOS only to mint a scoped **API Key** and then discards the WorkOS token, so the CLI follows the API-key scope model from [ADR 0060](./0060-cli-authentication-via-auth0-loopback.md). MCP uses WorkOS AuthKit/Connect tokens for the MCP root resource audience (`https://mcp.agent-paste.sh/`, with no-slash compatibility); their `scope` claim is authoritative and **Member-Only Scopes** remain unreachable. The invariant is unchanged: the same human is more powerful in the dashboard than through delegated agent surfaces.

The "implicitly granted every **Scope**" rule applies only to WorkOS AuthKit dashboard sessions resolved through `web` per [ADR 0059](./0059-web-app-session-and-auth-forwarding-to-api.md) and [ADR 0068](./0068-workos-authkit-for-web-app-auth.md). Delegated agent surfaces do not receive that grant: the CLI acts through a minted **API Key** from [ADR 0060](./0060-cli-authentication-via-auth0-loopback.md), and MCP acts through WorkOS AuthKit/Connect tokens for the MCP root resource audience per [ADR 0061](./0061-mcp-worker-with-oauth-only-via-auth0-dcr.md). For MCP tokens, the JWT `scope` claim is authoritative and the implicit grant does NOT apply. **Member-Only Scopes** are unreachable from non-dashboard surfaces regardless of the underlying Workspace Member's identity.

## Consequences

- One `requireScopes(scope[])` middleware serves every authenticated route in `api`. It accepts a WorkOS **Workspace Member** session, an MCP WorkOS bearer, or a scoped **API Key** and rejects when the route's required scopes are not a subset of the actor's granted scopes. For WorkOS dashboard sessions, the middleware applies the implicit member grant; for MCP WorkOS tokens, it reads `scope` from the JWT and does not apply the implicit grant.
- The auth context exposes `actor.type` (`member` or `api_key`) so handlers can branch when behavior depends on actor type without doing so for authorization checks. The audience (`aud=api/v1`, `aud=api/v1/cli`, `aud=mcp`) is recorded on the auth context for logging and rate-limit attribution but is not part of any glossary concept.
- `POST /v1/api-keys` rejects any attempt to grant a member-only **Scope** to a new key, closing the privilege escalation path where a captured key could mint a broader key. The runtime check is defense-in-depth; the grant-time check is the primary defense.
- A **Workspace Member** can call agent-targeted endpoints (**Publish**, **Access Link** lifecycle, **Display Metadata** updates) without minting an ephemeral **API Key**. Audit events record `actor_type='member'` and `actor_id` as the member id.
- **Creator** attribution records the member when an **Artifact** is created from a future dashboard flow and the **API Key** when created from an agent flow. CONTEXT.md already supports this dual form.
- `audit_events.actor_type` and `idempotency_records.actor_id` cover both shapes; no per-actor-type tables.
- CONTEXT.md has been updated: the **Scope** definition is generalized, **Member-Only Scope** is introduced as a glossary term, and the relationships block adds the member-implicit-grant and member-only-scope lines. Per the audience-scoped exception above, both the **Scope** and **Member-Only Scope** entries and the relationships block were further refined when [ADR 0060](./0060-cli-authentication-via-auth0-loopback.md) and [ADR 0061](./0061-mcp-worker-with-oauth-only-via-auth0-dcr.md) landed, so the dashboard/delegated-surface distinction is visible from the glossary.

## Considered Options

- Two parallel actor-typed middlewares (`requireMember`, `requireApiKey(scopes)`): explicit but doubles the auth surface, forces every route to opt into one path, and ambient member-overrides-everything is easy to add by accident.
- Unified scope model with member-only **Scopes**: one middleware, one rule, and the member-vs-key asymmetry expressed as data in the scope registry rather than as branching code.
