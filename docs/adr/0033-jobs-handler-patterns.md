# Jobs Handler Patterns: `runCommand` Gating, System Actors, and Post-Commit Sequencing

`jobs` handlers follow three intertwined rules: `runCommand` (ADR 0029) is required whenever a handler writes durable business state and skipped when it does not; every system actor is per-kind; Deletion and Retention handlers run a strict `runCommand` → KV denylist write → enqueue `byte-purge` sequence, with a single nullable timestamp column on `artifacts` and `revisions` providing crash recovery. This tightens ADR 0019's "use `runCommand` with an explicit system actor" by saying when not to, and replaces the textbook outbox pattern with token-exp-bounded recovery via ADR 0028.

## Considered Options

- **All handlers go through `runCommand`.** Maximally uniform, but wastes a Postgres transaction and an `idempotency_records` row on R2-only operations that are already naturally idempotent.
- **Outbox pattern for post-commit work.** Stronger guarantees against the gap between Postgres commit and KV write, but adds a durable outbox table, a drain worker, and lifecycle around it. Rejected because the signed-content-token natural exp (15 min per ADR 0028) already bounds the consistency window and the cron sweep is the natural recovery mechanism.
- **Single global `actor_id='jobs'`.** Less metadata to manage but loses the audit-log signal that distinguishes "Auto Deletion swept this artifact" from "Retention removed this revision."

## Consequences

- **`runCommand` gating.** A handler uses `runCommand` if and only if it writes durable business state to Postgres. Handlers that update Postgres go through it even when their audit-event list is empty (e.g. `bundle-generate` flips `revisions.bundle_status` without emitting an audit event per ADR 0019). Handlers that touch only R2 (every `byte-purge` flavor) skip it. `idempotency_records` GC also skips it because the table it deletes from is the wrapper's own bookkeeping; running it through `runCommand` would write a row to clean up rows.
- **Per-kind system actors.** All `actor_type='system'`, `actor_id` from `{auto_deletion, retention, upload_cleanup, safety_scan, bundle_generate}`. Audit logs read clearly and the `idempotency_records` composite key naturally segments by actor without collision.
- **Idempotency keys derive from target identity.** `(workspace_id, actor_id, operation, idempotency_key)` where `idempotency_key` is the target row's ID (`artifact_id`, `revision_id`, `session_id`), not the Queue message ID. Retries of the same enqueue, repeated sweep ticks, or distinct messages targeting the same row all collapse onto the same idempotency row. `safety-scan` includes `scanner_id` and `scanner_version` so a scanner upgrade re-scans without manual TTL expiry.
- **Post-commit sequencing for Deletion and Retention.** (1) `runCommand` writes Postgres state + audit event and commits. (2) KV denylist entries are PUT with retries (3 attempts, exponential backoff capped at ~1s). (3) `byte-purge` messages are enqueued. On persistent KV failure in (2), the handler does **not** enqueue purge and exits; the cron sweep re-discovers the target on the next tick.
- **Single recovery column.** `artifacts.bytes_purge_enqueued_at TIMESTAMPTZ NULL` and `revisions.bytes_purge_enqueued_at TIMESTAMPTZ NULL`. Set by a direct UPDATE outside `runCommand` after enqueue succeeds — a deliberate exception to "no Postgres writes outside `runCommand`" for an operational bookkeeping field with no audit semantics. The sweep query for recovery is `WHERE status IN ('deleted', 'retained') AND bytes_purge_enqueued_at IS NULL`.
- **No `bytes_purged_at`.** Purge handlers stay DB-free. Visibility into un-purged R2 bytes comes from R2 lifecycle metrics, not from the `artifacts` row.
- **Two intentional direct-UPDATE exceptions** to the "all DB writes go through `runCommand`" rule: the `bytes_purge_enqueued_at` set and the `idempotency_records` GC `DELETE`. Both are flagged with code comments. Every other Postgres write in `jobs` goes through `runCommand`.
- **Queue handlers re-check current state before acting.** `bundle-generate` and `safety-scan` read `revisions.status` and the parent `artifacts.status`; if the target is retained or deleted between enqueue and consume, the handler short-circuits without an error and without a `runCommand` invocation. A shared helper in `packages/commands/` enforces the pattern.
