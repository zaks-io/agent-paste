# Error Envelope and Generic-404 Boundary

Status: Accepted. Updated by [ADR 0047](./0047-access-link-signed-url-with-fragment-encoded-payload.md) to apply the generic boundary to Access Link Signed URL resolution instead of stored bearer-token routes.

Public REST errors use a fixed envelope `{ error: { code, message, docs?, request_id } }` where `code` is a stable snake_case identifier that is part of the `/v1` API contract. **Access Link Signed URL** resolution returns a generic `not_found` with status 404 for every failure mode (invalid signature, revoked, expired, locked down, retention-removed, deleted, wrong workspace) so callers cannot probe link state through error responses. Authenticated routes return specific reasons after **Workspace** authorization succeeds. Cross-workspace artifact references return `artifact_not_found` rather than `unauthorized` so workspace existence cannot be probed by a caller in a different **Workspace**. This makes ADR 0015's generic-failure rule concrete and extends it to a full error contract.

## Consequences

- Every response (success and error) echoes `X-Request-Id` as both a header and, on errors, an `error.request_id` field. The CLI surfaces `request_id` in `--json` output for support flows.
- `code` is enumerated in the OpenAPI document for each route. Adding new codes is backward-compatible; renaming or removing requires `/v2`.
- The status-code map is fixed: 400 for validation, 401 for unparseable or missing auth, 403 for `insufficient_scope`, 404 for `not_found` or `artifact_not_found`, 409 for `idempotency_in_flight` and state conflicts (`draft_revision_conflict`, `access_link_lockdown_active`), 410 for `artifact_deleted` and `revision_retained` on authenticated reads, 422 for semantic publish failures (`entrypoint_not_in_revision`, `render_mode_incompatible`), 429 for `usage_policy_exceeded`, `rate_limited_actor`, `rate_limited_workspace`, or `rate_limited_artifact`, 503 for `database_unavailable`.
- Access Link Signed URL resolution has a dedicated handler path: any internal failure (invalid signature, missing row, revoked, expired, lockdown active, deleted, retained, wrong workspace) becomes `{ error: { code: 'not_found' } }` with status 404. The actual reason is captured in the operational log with the resolved `access_link_id` where available so support can diagnose without leaking through the wire.
- Authenticated reads return the specific access-failure reason after workspace authorization succeeds: `artifact_deleted` (410), `revision_retained` (410), `artifact_locked` (403), `revision_unpublished` (404). Auth failure ahead of this still returns `unauthorized` or `insufficient_scope`.
- Workspace-existence leakage is closed at the query layer: a fetch for an artifact id that exists in a different **Workspace** returns the same `artifact_not_found` as one that does not exist anywhere. Repository helpers scope every read by `workspace_id` so route handlers cannot accidentally bypass the check.
- `message` is free-form, may include caller-relevant detail such as scope diffs (`Actor has scopes [write], needs [write, read, share]`), and is not part of the stability contract. Clients match on `code`, not `message` or HTTP status.
- `docs` is optional but populated for common codes, pointing at `https://agent-paste.sh/docs/errors/<code>`.
